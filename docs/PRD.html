01产品概述

1.1 项目背景

台球（尤其是三库台球与九球）实战中，玩家需熟练应用"颗星公式（Diamond System）"进行走位计算。目前缺乏脱离实体的线上专项脑力训练工具，玩家只能依赖线下球桌或碎片化视频学习，难以形成系统的心算肌肉记忆。

1.2 产品定位

CueMath 是一款面向移动端的高流畅度、强反馈 H5 训练应用，帮助玩家通过碎片化时间形成颗星公式的心算肌肉记忆。产品核心理念是：把抽象的几何公式，转化为"看得见轨迹、点得到答案"的即时反馈训练。

1.3 业务目标

填补市场空白

差异化

提供首个线上"颗星公式专项训练"工具，覆盖三库 / 九球走位心算。

提升实战精度

价值

通过即时轨迹反馈与错题数据，帮助玩家识别"算力盲区"，针对性补强。

社交裂变获客

增长

借"脑力测试报告海报"与好友排行榜，实现低成本社交传播。

02目标用户与使用场景

2.1 用户画像

P1 · 进阶玩家

核心

三库 / 九球进阶玩家，已掌握基础公式，需要高频心算训练以巩固走位精度。痛点：实战中算不准、脱离球桌无法练习。

P2 · 台球新手

潜力

颗星公式零基础，需要从固定点位的简单场景入门。痛点：公式晦涩、缺交互式教学。

P3 · 竞技挑战者

高频

追求极限心算速度与连击成就感。痛点：现有工具无计时压榨模式与排行榜。

2.2 用户痛点汇总

  • 脱离实体球桌无法训练走位心算——线下场地与时间成本高。
  • 公式抽象难以理解——缺乏"输入公式 → 看见轨迹"的直观反馈。
  • 不知算力盲区在哪——缺乏错题数据可视化与针对性复盘。

2.3 典型使用场景

通勤 / 排队等碎片时间（5–10 分钟）：刷一组限时题保持心算手感；
球局开始前热身：提前激活颗星公式心算反应；
赛后复盘：针对错题区间进行专项训练。

编号用户故事优先级

03核心功能需求

3.1 核心状态机（Game Loop）

单局训练由以下五个状态顺次流转，构成 Core Loop：

图 3-1　核心状态机：Init → Generate → Wait_Input → Animation → Settle，Animation 处分命中 / 错误两支

状态说明

  • Init：加载球桌素材、音效，初始化刻度系。
  • Generate：根据当前模式（新手 / 进阶 / 大师）从题库策略中生成一组 (A, B) 坐标，渲染母球与目标球。
  • Wait_Input：开启库边热区监听，等待用户点击第一库目标点 C_user。
  • Animation：锁死交互。计算正确点 C_real。
    • 若 C_user == C_real，播放命中动画及特效。
    • 若 C_user != C_real，播放错误路线动画，随后展示正确轨迹与公式面板。
  • Settle：更新得分、连击数（Combo）、记录耗时与错题，进入下一题（返回 Generate）。

3.3 颗星公式与轨迹算法

颗星公式是本产品的计算核心，公式表达如下：

A（出发点） − B（目标点）  =  C（第一库瞄准点）

算法策略采用基于几何映射的伪物理计算，刻意不引入 Box2D 等物理引擎，以保证轨迹可预测、动画轻量：

  • 状态分离：球桌视为 2D 坐标系，球与库的关系由映射字典描述。
  • 路径生成：预先建立颗星刻度到二维坐标的映射字典；根据公式计算出多段折线的端点坐标 (x1, y1) → (x2, y2) → (x3, y3)。
  • 动画执行：使用 Tween 动画库控制母球沿生成的折线序列按匀速 / 线性减速平移。

3.4 辅助功能

线索盲抽开关

Phase 2

可选择隐藏 / 显示公式提示，逐步脱离依赖，强化纯心算能力。

公式解析面板

Phase 1

答题后展示完整公式拆解（A / B / C 取值与折线坐标），帮助复盘。

音效反馈

Phase 2

引入音效引擎，命中 / 错误 / 连击各有专属反馈。

算力盲区报告

Phase 2

训练结束后基于错题矩阵高亮权重区，输出个人盲区热力图。

3.5 关键功能验收标准

查看 MVP（Phase 1）核心功能验收标准
  • AC-01 状态机闭环：从 Init 出发可顺次走完全部 5 状态并回到 Generate，无卡死。
  • AC-02 库边点击：点击任意颗星热区，系统可在 100ms 内进入 Animation。
  • AC-03 命中判定：当 |C_user - C_real| ≤ 容差 时判定命中，播放绿色实线轨迹 + 命中特效。
  • AC-04 错误反馈：未命中时播放红色虚线错误路线，随后叠加绿色正确轨迹与公式面板。
  • AC-05 连击累计：连续答对时 Combo +1，错误归零，并在 Settle 阶段同步得分。
  • AC-06 进阶模式通关：连续答对 15 题触发晋级，记录可写入 Local Storage。

4.1 视图布局（竖屏 60 / 40）

采用竖屏模式，上下两区分别承担"看球桌"与"操作控制"两个心智：

  • 上半屏 60%：二维球桌区域（视角自上而下）。
  • 下半屏 40%：控制台（得分、计时器、辅助选项、公式解析板）。

4.2 球桌示意图与库边热区

图 4-1　球桌俯视图：颗星刻度（0–50）、第一库热区高亮、母球 A、目标球 B、正确 / 错误轨迹示例

4.3 交互热区规范

🎯 库边直选（核心交互方案）
第一库对应的颗星点（0–50）设置不可见的点击热区。热区宽度需适配手指点击（建议单点热区宽度 ≥ 44px），防止误触；相邻热区之间留出安全间距。

4.4 UI 元素规范

元素规范
球桌极简扁平风（Flat Design）或拟物绿呢台布；刻度数字字号需保证移动端清晰可读（≥ 12px）。
正确轨迹绿色，实线，透明度 0.8。
错误路线红色，虚线，透明度 0.6。
命中反馈播放命中动画及特效（粒子 / 高光）。
错误反馈播放错误路线动画，随后叠加正确轨迹与公式面板。

⚙️ 关键设计决策：伪物理而非物理引擎
刻意不引入 Box2D 等物理引擎。理由：颗星训练目标是"公式心算"，轨迹必须可预测、可复现，物理碰撞的随机性反而干扰公式学习；同时伪物理计算更轻量，适配移动端性能预算。

06数据与存储

6.1 本地存储（Local Storage）

所有用户数据均存储于浏览器本地，不上传服务端（Phase 3 排行榜除外）：

  • 最高分：各模式历史最高得分。
  • 解锁状态：新手 / 进阶 / 大师 / 限时模式的解锁进度。

07非功能需求

⚡ 性能

动画渲染 60fps（PixiJS WebGL）；首屏加载 ≤ 2s（4G 网络）；单题动画响应 ≤ 100ms。

📱 兼容性

iOS Safari、Android Chrome、微信 webview；适配 360px–414px 主流移动宽度；横屏时提示锁定竖屏。

🔒 安全

本地数据不上传，保护隐私；限时模式做前端时间篡改校验；Phase 3 排行榜需服务端二次校验防作弊。

♿ 可访问性

刻度数字对比度 ≥ AA；支持系统字体放大；关键操作提供非颜色反馈（震动 / 音效）。

🧩 可维护性

状态机与算法模块化，便于扩展新模式；题库策略与渲染解耦，算法层可独立测试。

📦 体积

H5 首屏资源（gzip）目标 ≤ 500KB，PixiJS 按需加载，素材采用 SVG / WebP 压缩。

A.1 术语表

台球与颗星公式相关术语
术语释义
颗星（Diamond）球桌库边的刻度标记点，是颗星公式的计算基准。
母球（Cue Ball）被球杆击打的白色球，公式中记作 A（出发点）。
目标球（Object Ball）需要被击中的球，公式中记作 B（目标点）。
库（Cushion / Rail）球桌的边框；"一颗星"= 撞一次库。
三库（Three-cushion）母球需撞击库边三次再命中目标球的玩法。
颗星公式（Diamond System）利用颗星刻度进行走位计算的几何公式：A − B = C。
Combo（连击）连续答对的次数，用作得分加成与成就感反馈。
